五、ASR 與音檔處理工具

1. ffmpeg

用途：
切分 Podcast 長音檔，避免單檔超過 ASR API 上傳限制。

目前使用：
已安裝並用於將 EP667 音檔切成 4 個 MP3 chunk。

2. OpenAI Speech-to-Text API

官方文件：
https://platform.openai.com/docs/guides/speech-to-text

用途：
將音檔 chunk 轉成逐字稿。

目前設計：
scripts/transcribe_openai.py 預設使用 gpt-4o-mini-transcribe，呼叫 /v1/audio/transcriptions。

限制：
單檔上傳限制為 25 MB，因此長音檔需先透過 scripts/split_audio.py 切段。

目前狀態：
轉錄腳本已建立；本機曾嘗試使用 OPENAI_API_KEY 呼叫 API，但回傳 insufficient_quota。第一階段目前改以 whisper.cpp 本機轉錄為主要路線。

本機設定方式：

1. 建立 .env 檔案。
2. 寫入 OPENAI_API_KEY=你的 key。
3. .env 已由 .gitignore 排除，不會推到 GitHub。

範例：
可參考專案根目錄的 .env.example。

注意：
不要把真實 API key 寫入 README、docs、CSV 或任何會 commit 的檔案。
.env 裡的值必須是真正的 OpenAI API key，通常以 sk- 開頭；不要保留 .env.example 裡的範例文字。

3. 合併逐字稿

工具：
scripts/combine_transcripts.py

用途：
把同一集的多個 chunk transcript 合併為 data/raw/transcripts/{episode_id}.txt。

注意：
逐字稿原文仍屬本機 raw artifact，不進 Git。

4. whisper.cpp

工具：
whisper-cli

安裝方式：
brew install whisper-cpp

模型：
data/models/ggml-base.bin

模型來源：
https://huggingface.co/ggerganov/whisper.cpp/resolve/main/ggml-base.bin

用途：
在 OpenAI API 配額不足或不想使用雲端轉錄時，使用本機 Whisper 產生逐字稿。

目前使用：
已用 scripts/transcribe_whisper_cpp.py 將 EP667 的 4 個 audio chunks 轉成逐字稿，並用 scripts/combine_transcripts.py 合併成 data/raw/transcripts/ep_20260603_667.txt。

注意：
模型檔、chunk transcript 與合併逐字稿皆為本機 raw artifact，由 .gitignore 排除，不進 Git。
原始音檔與切段音檔只作為轉錄暫存素材；確認合併逐字稿已產生、metadata 已更新後，即可刪除對應音檔與 chunks。

5. 逐字稿候選標的搜尋

工具：
scripts/search_transcript_mentions.py

用途：
讀取 data/processed/instruments.csv 的 name、ticker 與 aliases，掃描 data/raw/transcripts/ 下的合併逐字稿，輸出命中的 episode、行號、標的與上下文片段。

範例：
python3 scripts/search_transcript_mentions.py --episode-id ep_20260530_666 --episode-id ep_20260527_665 --limit-per-alias 2

注意：
此工具只用來加速人工初篩，不會自動新增 approved mention。正式進入回測前仍需人工校對 ASR 文字與上下文。

6. mention 人工審核工作表

工具：
scripts/build_review_queue.py

用途：
讀取 data/processed/mentions.csv 中 review_status 為 pending 的正式 mention，產生 reports/pending_mentions_review.csv。輸出包含短 evidence、逐字稿行號提示、短上下文，以及人工要填寫的 review_decision 與 corrected_* 欄位。

範例：
python3 scripts/build_review_queue.py

套用審核結果：
python3 scripts/apply_review_decisions.py --dry-run
python3 scripts/apply_review_decisions.py

注意：
review_decision 空白時不會回寫。正式套用前先跑 --dry-run，確認沒有不合法的 review_decision、stance、conviction 或 time_horizon。

7. 報酬檢查報表

工具：
scripts/build_return_report.py

用途：
讀取 data/processed/mentions.csv、data/processed/episodes.csv 與 data/processed/mention_returns.csv，輸出 reports/approved_company_bullish_returns.csv。此報表只列出已 approved、mention_type=company、stance 為 bullish / bearish、且 ticker 不空白的正式 mentions。

範例：
python3 scripts/build_return_report.py

目前使用：
已產生第一版 reports/approved_company_bullish_returns.csv，共 5 筆 approved company bullish mentions。其中 1 筆已有 7d return，其餘因 horizon 價格尚未到期而維持 pending_price。

注意：
此報表是 mention_returns.csv 的人工檢查視圖。每次重跑 scripts/fetch_prices.py 更新價格與報酬後，應再重跑 scripts/build_return_report.py，讓可讀報表同步反映最新 horizon return。
