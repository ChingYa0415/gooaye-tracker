版本：0.1
日期：2026-06-05

一、文件目的

本文件定義 data/processed/ 目錄下核心 CSV 的欄位、型別、允許值與使用規則。

核心 CSV：

1. episodes.csv
   記錄節目、影片、文章或社群貼文等來源單位。

2. mentions.csv
   記錄從來源內容中抽出的公司、股票、產業或主題觀點。

3. instruments.csv
   記錄股票、指數、產業概念或其他可追蹤標的。

4. mention_returns.csv
   記錄每筆 approved mention 的報酬計算結果。

輔助 CSV：

1. transcript_inputs.csv
   記錄待轉錄音檔、來源 episode 與預計逐字稿輸出路徑。

2. audio_downloads.csv
   記錄已下載到本機的音檔路徑、大小與 hash。音檔本身不進 Git。

3. audio_chunks.csv
   記錄已切分的音檔段落。chunk 檔案本身不進 Git。

4. transcription_runs.csv
   記錄 ASR 轉錄執行結果。逐字稿原文不進 Git。

5. data/prices/*.csv
   記錄 scripts/fetch_prices.py 抓取的每日價格資料。

報表與人工工作表：

1. reports/pending_mentions_review.csv
   記錄待人工校對的 pending mentions、短證據、逐字稿行號提示與審核決策欄位。

2. reports/approved_company_bullish_returns.csv
   記錄已審核、可回測的單一公司 bullish / bearish mentions，以及目前可取得的報酬結果。

3. reports/summary.csv
   記錄目前資料管線的總覽指標，例如 episode 數、mention review 狀態、return 狀態、可得 horizon 與平均報酬。

本機設定檔：

1. .env
   用於儲存 OPENAI_API_KEY 等本機密鑰。此檔案已被 .gitignore 排除。

2. .env.example
   範例格式，可 commit，但不得包含真實 key。

第一版採 CSV，是為了快速建立可手動操作的 MVP。未來若資料量增加，可轉成 SQLite 或 PostgreSQL。


二、通用規則

1. ID 規則
   - episode_id 使用 ep_ 前綴，範例資料使用 sample_ep_ 前綴。
   - mention_id 使用 m_ 前綴，範例資料使用 sample_m_ 前綴。
   - instrument_id 使用 inst_ 前綴。

2. 日期格式
   - 日期使用 YYYY-MM-DD。
   - 若未來需要時間，另增 published_time 或 published_at_time，不在日期欄混用。

3. 空值規則
   - 未知但未來可補：留空。
   - 無此欄位意義：留空。
   - 判斷不明：使用 unclear。
   - 市場未知：使用 unknown。

4. 布林值
   - 使用 true / false 小寫文字。

5. 範例資料
   - sample_ 開頭的資料只用於驗證欄位。
   - 正式統計或報表應排除 sample_ 開頭的資料。


三、episodes.csv

用途：
記錄每一個來源單位。一集 Podcast、一支 YouTube 影片、一篇文章或一則社群貼文都算一筆 episode。

欄位：

1. episode_id
   型別：string
   必填：是
   說明：來源單位唯一 ID。
   範例：ep_20260605_001

2. source_name
   型別：string
   必填：是
   說明：節目或來源名稱。
   範例：股癌 Gooaye

3. source_type
   型別：enum
   必填：是
   允許值：podcast / youtube / threads / article / manual
   說明：來源類型。

4. title
   型別：string
   必填：是
   說明：節目、影片、文章或貼文標題。

5. published_at
   型別：date
   必填：是
   說明：來源發布日期。

6. url
   型別：string
   必填：否
   說明：來源 URL。

7. duration_seconds
   型別：integer
   必填：否
   說明：音訊或影片長度秒數。文章或 Threads 可留空。

8. transcript_status
   型別：enum
   必填：是
   允許值：missing / manual / imported / generated / reviewed
   說明：逐字稿取得狀態。

9. notes
   型別：string
   必填：否
   說明：補充說明。


四、mentions.csv

用途：
記錄某一段內容中形成的股票、公司、產業或主題觀點。

欄位：

1. mention_id
   型別：string
   必填：是
   說明：mention 唯一 ID。
   範例：m_20260605_001

2. episode_id
   型別：string
   必填：是
   說明：對應 episodes.csv 的 episode_id。

3. published_at
   型別：date
   必填：是
   說明：來源發布日期，用於回測基準日。

4. company_or_theme
   型別：string
   必填：是
   說明：公司、股票、產業、指數或主題名稱。

5. ticker
   型別：string
   必填：否
   說明：股票代號。概念或產業可留空。

6. market
   型別：enum
   必填：是
   允許值：TWSE / TPEx / US / HK / concept / unknown
   說明：市場或標的類型。

7. mention_type
   型別：enum
   必填：是
   允許值：company / industry / theme / index / commodity
   說明：mention 標的類型。

8. stance
   型別：enum
   必填：是
   允許值：bullish / bearish / neutral / watch / past_review / unclear
   說明：方向性判斷，定義見 docs/標註規則.txt。

9. conviction
   型別：enum
   必填：是
   允許值：low / medium / high
   說明：觀點強度。

10. time_horizon
   型別：enum
   必填：是
   允許值：short / medium / long / unclear
   說明：觀點時間尺度。

11. evidence_text
   型別：string
   必填：是
   說明：支持標註的原文短句。approved mention 不可空白。

12. rationale
   型別：string
   必填：是
   說明：人工或 AI 對標註原因的簡述。

13. review_status
   型別：enum
   必填：是
   允許值：pending / approved / rejected / needs_context
   說明：人工審核狀態。

14. reviewer_note
   型別：string
   必填：否
   說明：審核補充說明。

正式回測納入條件：

1. review_status = approved。
2. stance = bullish 或 bearish。
3. evidence_text 不可空白。
4. 若 market 不是 concept，ticker 不可空白。
5. mention_id 不可以 sample_ 開頭。

六、mention_returns.csv

用途：
記錄每筆 mention 在指定期間的報酬與相對 benchmark 表現。

欄位：

1. mention_id
   型別：string
   必填：是
   說明：對應 mentions.csv 的 mention_id。

2. ticker
   型別：string
   必填：否
   說明：股票代號。概念型 mention 可留空。

3. market
   型別：enum
   必填：是
   允許值：TWSE / TPEx / US / HK / concept / unknown

4. stance
   型別：enum
   必填：是
   允許值：bullish / bearish
   說明：主要回測只處理 bullish / bearish。

5. base_trade_date
   型別：date
   必填：否
   說明：發布後第一個交易日。

6. base_price
   型別：number
   必填：否
   說明：基準日收盤價。

7. return_7d
   型別：number
   必填：否
   說明：7 天後報酬率。使用小數，例如 0.05 代表 5%。

8. return_30d
   型別：number
   必填：否
   說明：30 天後報酬率。

9. return_90d
   型別：number
   必填：否
   說明：90 天後報酬率。

10. return_180d
   型別：number
   必填：否
   說明：180 天後報酬率。

11. benchmark_return_7d
   型別：number
   必填：否
   說明：benchmark 7 天後報酬率。

12. benchmark_return_30d
   型別：number
   必填：否
   說明：benchmark 30 天後報酬率。

13. benchmark_return_90d
   型別：number
   必填：否
   說明：benchmark 90 天後報酬率。

14. benchmark_return_180d
   型別：number
   必填：否
   說明：benchmark 180 天後報酬率。

15. excess_return_7d
   型別：number
   必填：否
   說明：return_7d - benchmark_return_7d。

16. excess_return_30d
   型別：number
   必填：否
   說明：return_30d - benchmark_return_30d。

17. excess_return_90d
   型別：number
   必填：否
   說明：return_90d - benchmark_return_90d。

18. excess_return_180d
   型別：number
   必填：否
   說明：return_180d - benchmark_return_180d。

19. calculation_status
   型別：enum
   必填：是
   允許值：pending_price / calculated / missing_price / not_applicable / error
   說明：報酬計算狀態。

20. notes
   型別：string
   必填：否
   說明：錯誤原因或補充說明。

第一版計算規則：

1. 只計算 review_status=approved 的 mention。
2. 只有 mention_type=company、stance 為 bullish / bearish，且 ticker 不空白的 mention 會進入價格回測。
3. concept、watch、past_review 等 mention 會在 mention_returns.csv 中標為 not_applicable。
4. base_trade_date 使用 published_at 後第一個可取得的交易日。
5. 7 / 30 / 90 / 180 天報酬使用 base_trade_date 加上日曆天後第一個可取得的交易日。
6. 報酬使用 adj_close 計算；若來源沒有 adj_close，則使用 close。
7. benchmark 第一版：TWSE 使用 ^TWII，US 使用 ^IXIC。
8. 若未來 horizon 價格尚未出現，calculation_status 標為 pending_price。

十一、transcription_runs.csv

用途：
記錄 ASR 轉錄執行結果。逐字稿文字檔儲存在 data/raw/transcripts/，並由 .gitignore 排除。

欄位：

1. episode_id
   型別：string
   必填：是
   說明：對應 episodes.csv 的 episode_id。

2. chunk_index
   型別：integer
   必填：是
   說明：對應 audio_chunks.csv 的 chunk_index。

3. chunk_path
   型別：string
   必填：是
   說明：送入 ASR 的本機 chunk 路徑。

4. transcript_path
   型別：string
   必填：是
   說明：ASR 產生逐字稿的本機路徑。

5. model
   型別：string
   必填：是
   說明：ASR 模型。OpenAI 路線預設 gpt-4o-mini-transcribe。

6. transcription_status
   型別：enum
   必填：是
   允許值：generated / error
   說明：轉錄狀態。

7. notes
   型別：string
   必填：否
   說明：補充說明。

合併規則：

1. 每個 chunk 轉錄完成後，執行 scripts/combine_transcripts.py。
2. 合併後輸出 data/raw/transcripts/{episode_id}.txt。
3. 合併後逐字稿原文不進 Git，只作為人工標註與 AI 抽取的本機輸入。
4. 合併逐字稿確認存在後，原始音檔與切段音檔可視為可重建暫存檔，允許從本機刪除；Git 仍保留下載、切段與轉錄 metadata 供追蹤。

十三、approved_company_bullish_returns.csv

用途：
記錄由 scripts/build_return_report.py 產生的第一版報酬檢查報表。此報表從 mentions.csv、episodes.csv 與 mention_returns.csv 派生，用來人工檢查 approved company bullish / bearish mentions 的基準價、可得 horizon 報酬與相對 benchmark 超額報酬。

欄位群組：

1. mention 與 episode 欄位
   欄位：mention_id、episode_id、episode_title、published_at、company_or_theme、ticker、market、stance、conviction、time_horizon
   說明：用於追溯來源與原始標註。

2. 基準價欄位
   欄位：base_trade_date、base_price
   說明：發布後第一個可得交易日與該日價格。

3. 報酬狀態欄位
   欄位：available_horizons、calculation_status、return_notes
   說明：列出目前已可計算的 horizon，例如 7d；若後續價格尚未到期，calculation_status 會維持 pending_price。

4. 證據欄位
   欄位：evidence_text
   說明：保留原始短證據，方便人工檢查該 mention 是否真的值得納入回測。

5. horizon 報酬欄位
   欄位：return_7d、return_30d、return_90d、return_180d，以及對應的 _pct 欄位
   說明：標的自身報酬；小數欄用於後續程式計算，_pct 欄用於人工閱讀。

6. benchmark 與超額報酬欄位
   欄位：benchmark_return_*、excess_return_*，以及對應的 _pct 欄位
   說明：benchmark 報酬與標的相對 benchmark 的超額報酬。

使用規則：

1. 此報表是派生檔，可由 scripts/build_return_report.py 重建。
2. 不在此報表中手動修改 mention 判斷；若要修正，應回到 mentions.csv 或 review workflow。
3. 每次重跑 scripts/fetch_prices.py 更新 mention_returns.csv 後，應再重跑 scripts/build_return_report.py。


十四、summary.csv

用途：
記錄由 scripts/build_summary_report.py 產生的資料管線總覽。此報表用於快速回答目前有多少資料、審核狀態如何、多少 mention 可回測、目前已可計算哪些 horizon，以及目前可得報酬的平均表現。

欄位：

1. section
   型別：string
   必填：是
   說明：指標群組，例如 episodes、mentions、returns、reports、performance。

2. metric
   型別：string
   必填：是
   說明：指標名稱，例如 formal_total、review_status.approved、7d.avg_return。

3. value
   型別：number
   必填：是
   說明：機器可讀的原始數值。百分比以小數表示，例如 0.0392 代表 3.92%。

4. display_value
   型別：string
   必填：是
   說明：人工閱讀用顯示值，例如 3.92% 或 24。

5. notes
   型別：string
   必填：否
   說明：指標補充說明。

使用規則：

1. 此報表是派生檔，可由 scripts/build_summary_report.py 重建。
2. 每次更新 mention_returns.csv 或 approved_company_bullish_returns.csv 後，應重建 summary.csv。
3. Dashboard 或後續分析可直接讀取 section + metric 作為穩定指標鍵。


十五、下一步檢查清單

建立或修改資料後，至少檢查：

1. CSV 欄位數是否與資料字典一致。
2. sample_ 開頭資料是否被排除在正式統計之外。
3. approved mention 是否都有 evidence_text。
4. bullish / bearish 且非 concept 的 mention 是否有 ticker。
5. mention_returns.csv 是否只含正式回測候選資料或明確標記 sample。
6. instruments.csv 是否能支援常見別名查找。
7. transcript_inputs.csv 是否每筆都有 audio_url 與 transcript_path。
8. audio_chunks.csv 的 chunk_bytes 是否都低於 ASR 單檔上傳限制。
9. pending_mentions_review.csv 若有 review_decision，是否皆為允許值。
10. mention_returns.csv 中 calculated / pending_price rows 是否都有 base_trade_date 與 base_price。
11. data/prices/*.csv 是否每列欄位數一致。
12. approved_company_bullish_returns.csv 是否只包含 approved、company、bullish / bearish 且有 ticker 的正式 mentions。
13. 每次更新價格後，是否已同步重建 approved_company_bullish_returns.csv。
14. summary.csv 是否包含 episodes.formal_total、mentions.formal_total、returns.formal_total 與 reports.approved_company_bullish_returns_rows 等核心指標。
